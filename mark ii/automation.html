Automation

If you're doing it more than twice, automate it

Automation is about freeing your team to do work that actually matters, and giving them leverage to do exponentially more... eliminating toil.

The trap

Developers love the new shiny. Give someone an automation problem and there's a good chance they'll disappear for a week, emerge with something clever, and watch it fall apart on first contact with real data.

It's like that scene in Gung Ho where Michael Keaton gets into a car and it falls apart after two steps. Edge cases matter. Raw data is always full of edge cases. And nobody thinks about edge cases when they're excited about the elegant solution they're building.

The fix isn't better planning. It's starting smaller and iterating often. Build something that solves the immediate problem. Run it against real data. Watch it break. Fix it. Repeat. This only works if you have a CI/CD pipeline that accommodates fast iteration. If deploys are painful, people batch up changes, and batched changes hide problems.

Lego bricks, not unicorns

My Unix background shaped how I think about automation. Small tools that do one thing well. Pipes that connect them. Simple systems that bolt together to create complex patterns.

Don't build a unicorn that solves your specific problem in a clever way. Build lego bricks that snap together. The brick you build today to solve one problem might solve three other problems next month. The unicorn just sits there, doing its one trick, until someone has to maintain it and realizes they can't.

This ties directly to platform engineering. Good automation becomes part of the platform. Bad automation becomes tribal knowledge that only one person understands.

When not to automate

The threshold is gut feel at this point. If I'm doing something more than twice, I look for a way to script it. Unless scripting takes ten times the effort of just doing the task.

But effort isn't the only factor. The real question is: does this free up the team? If I can spend a day automating something and it gives five people their afternoons back, that math works. If I'm just scratching my own itch, maybe the manual way is fine.

Don't spend time polishing a turd. Sometimes good enough is perfect for getting through a problem and moving on.

The case against toil

Toil is work that's manual, repetitive, automatable, and scales linearly with system size. It's the stuff that makes you feel busy but doesn't make you better.

Every hour spent on toil is an hour not spent on improvements. Good automation compounds. It pays dividends every time it runs.

"The first rule of any technology used in a business is that automation applied to an efficient operation will magnify the efficiency. The second is that automation applied to an inefficient operation will magnify the inefficiency."

But there's a catch: automation applied to an inefficient process just magnifies the inefficiency. Before automating something, make sure it's worth automating.

What good automation looks like

  • Idempotent: Run it once or run it ten times, same result.
  • Observable: You can see what it did and why.
  • Reversible: When possible, you can undo it.
  • Documented: The next person (including future you) can understand it.
  • Tested: You know it works before it runs in production.

Tools of the trade

  • Python (primary language)
  • CI/CD pipelines
  • API integrations (REST, FuelSDK)
  • Scripting for cloud infrastructure
