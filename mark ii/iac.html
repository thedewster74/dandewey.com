Infrastructure as Code

Your infrastructure should be in version control

IaC means treating infrastructure the way we treat software: versioned, tested, reviewed, and repeatable.

Why it matters

Manual infrastructure is a liability. It's inconsistent, undocumented, and impossible to reproduce. When the person who set it up leaves, so does the knowledge.

Infrastructure as Code changes that. Your infrastructure becomes:

  • Documented — The code IS the documentation
  • Reviewable — Changes go through pull requests
  • Reproducible — Spin up identical environments on demand
  • Auditable — Git history shows who changed what and when

The difference

❌ Without IaC

"We think Dave set up the firewall rules. He left six months ago. Nobody knows why port 8443 is open."

✓ With IaC

"Port 8443 was opened in commit abc123 for the legacy API migration. Here's the PR with the discussion."

My approach

  • Declarative over imperative — Describe what you want, not how to get there
  • Modules for reuse — Don't repeat yourself across environments
  • State management — Know what exists and what should exist
  • Drift detection — Catch when reality diverges from code (see: MonitorMatrix)

Related work

MonitorMatrix, my configuration-verification project, is directly tied to IaC principles — it detects when deployed infrastructure drifts from its intended state, closing the loop between "infrastructure as code" and "infrastructure as deployed."
